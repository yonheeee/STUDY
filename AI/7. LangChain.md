# 07. LangChain

## 1. Post-traing(Instruction-tuning, RLHF)
### 1. Pretaining vs Posting-training
**거대 언어 모델 학습 패러다임**
<br>
- `Pre-traing(사전학습)` : 방대한 텍스트 데이터를 통해 언어와 지식을 배우는 단계<br>
- `Post-train(사후학습)` : 사람이 원하는 방식으로 대화하고, 안전하고, 유용하게 만드는 단계
<br>
<br>

**Pre-traing(사전학습)**
<br>
방대한 텍스트 데이터를 이용한 Self-supervised learning을 통해 언어 패턴, 지식등을 배움<br>
학습목표 : **다음 단어 예측(Next Token Prediction)**<br>
**Post-training(사후학습)**
<br>
유저의 의도를 파악하고 원하는 답변이 모델이 응답하도록 사후 학습 진행<br>
대표 기법 : Instruction-tuning, RLHF, DPO, RLVR
<br><br>


### 2. Instruction-tuning
사전 학습 후 거대 언어 모델은 유저의 의도와 일치X -> 파인튜닝 진행
<br>
 `파인 튜닝` : 이미 학습된 기존 모델에 특정 작업이나 모델에 맞게 **추가로 학습**
<br>
`Instruction-tuning`: 언어모델이 사람이 내린 지시문(instruction)을 따르도록 학습하는 단계<br>
- 정답 레이블이 요구되며, 다양한 태스크를 풀 수 있도록 적응하도록 중점 
- (지시문, 응답) 쌍을 모아서 언어모델을 파인튜닝
- 새로운(모델이 학습하지 않은) 태스크에서 평가를 진행

ex) Super-Natrullnstructions(데이터 셋은 1.6k+ 태스크와 3M+),<br>
    MMLU(Masssive Multitask Language Understanging, 57개의 지식을 요구하는 태스크에서 언어모델의 성능 평가하는 벤치마크)
<br>
**->Instruction-tuning에는 Pre-training에 비해 많은 데이터 필요 X**
<br>
<br>
### 3. Reinforcemen Learning from Human Feedback(RLHF)
**Instruction-tuning한계**<br>
1. 개방형/창의적 생성과 같은 태스크 정답 존재X
2. 언어 모델링은 모든 토큰 레벨 오류를 동일 취급(오류마다 다르게 작용)
3. 사람이 만든 답변(정답 레이블) 최적 아닐 수 있음

**파이프라인 큰 그림**<br>
`입력 -> LLM -> 출력 -> 사람들 평가 -> 데이터 비교
 -> 학습 -> reward model(보상) -> 강화학습 -> LLM`

**인간의 선호도 모델링**
<br>
문제 : 인간의 판단 일관성 떨어짐, 기준 어긋<br>
해결 방법 : 직접 점수X, 응답 비교<br>
해당 데이터로 리워드 모델(reward model) 학습<br>
**장단점**<br>
장점 : pre-training과 instruction-tuning보다 더 나은 성능 제공<br>
주의점 : 리워드 모델이 잘 작동하는지 먼저 확인해야함<br>
ChatGPT : Instruction-tuning + RLHF

### 4. Next
**리워드 모델 한계**<br>
인간의 선호도 일관성 부족(Reward Hacking은 강화학습에서 자주 발생), Hallucination(환각) 발생 <br>
**DPO**<br>
`Direction Preference Optimization(DPO)`: RLHF에서 'RL(강화학습)'을 제거, 사람의 선호 데이터 직접 확률 최적화<br>
**RLVR**<br>
`Reinforcement Learning with Verifiable Reward(RLVR)`: 인간 피드백 대신 정답 검증이 가능한 문제에서 자동으로 보상

## 2. Retrieval-augmented Language Model(Information Retrieval, RAG)
### 1. Basic Concepts
`Retrieval-augmented LM`: 추론 시 외부 데이터 저장소를 불러와 활용하는 언어모델(Datastore, Query, Index, Language Model로 구성)<br>
`Retrieval(rjator)`: Datastore에 있는 수 많은 정보 중, 주어진 쿼리와 가장 관련성이 높은 정보를 찾아내는 과정
`**Retrieval-augmented Generation(RAG)**`: 사용자의 질문에 답하기 위해, datastore에 관련된 정보를 검색(retrieval)해와서, 이를 언어모델이 생성 단계에 함께 활용하는 방법
<br><br>
### 2. Information Retrieval
`Information Retrieval` : 사용자의 질의(Query)에 맞는 정보를 대규모에서 찾아 제공(가장 관련성 높은 정보 제공)<br>
**종류**
- `Sparse Retriever(어휘적 유사도 기반)` : 쿼리와 문서간 정확한 용어 일치(어휘적 유사도)에 기반<br>
ex) TF-IDF,BM-25
- `Dense Rereiever(의미적 유사도)` : 퀴리와 문서를 표현하기 위해 dense vector을 활용해 의미적 유사도 기반<br>
ex) DPR, Contriever,Openai-embedding, etc

<br>

**Sparse Retriever - TF-IDF**<br>
문서 내 특정 단어의 중요도를 나타내는 가중치 방식
- `TF(Term Frequency)` : 단어가 문서에 얼마나 자주 등장하는지 -> 자주 등장하는 단어는 중요
- `IDF(Inverse Document Frequency)` : 단어가 전체 코퍼스에 얼마나 드물게 등장하는지 -> 너무 많은 문서에 등장하는 단어 덜 중요(ex. and, a, is)
